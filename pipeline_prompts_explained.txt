================================================================================
ПОЯСНЕНИЕ PIPELINE ПРОМПТОВ (Quiz Streamlit App)
================================================================================
Документ описывает последовательность вызовов LLM в quiz_streamlit_app.py и связь
с файлами промптов. Подробные параметры моделей см. quiz_prompts_config.py
(словарь PROMPT_CONFIGS, функция get_prompt_config).

Общие условия:
- Все вызовы идут через OpenRouter (OpenAI-совместимый API), если не указано иное.
- Ключ API: OPENROUTER_API_KEY (st.secrets или переменная окружения).
- Модели для каждого шага можно менять в сайдбаре Streamlit или через env
  (см. раздел «Переменные окружения» ниже).


--------------------------------------------------------------------------------
СХЕМА ЭТАПОВ (по ходу прохождения квиза)
--------------------------------------------------------------------------------

  [Настройка] → payload JSON, язык, сложность
         │
         ▼
  ┌──────────────────┐
  │ 1. GENERATE      │  Генерация списка вопросов (SHORT ANSWER или REASONING)
  └────────┬─────────┘
           ▼
  ┌──────────────────┐
  │ 2. REPHRASE      │  Для каждого экрана вопроса: разговорная подача текста
  │    (на вопрос)   │  (кеш в session_state.rephrase_cache)
  └────────┬─────────┘
           ▼
  Пользователь отвечает → Submit
           │
           ▼
  ┌──────────────────┐
  │ 3. VALIDATE      │  Оценка ответа: баллы, intent, validation_error (JSON)
  └────────┬─────────┘
           │
     ┌─────┴─────┐
     │ прошёл    │  → следующий вопрос (снова шаг 2)
     │ порог     │
     └─────┬─────┘
           │ не прошёл
           ▼
  ┌──────────────────┐
  │ 4. FOLLOW-UP     │  Отдельный «тьюторский» текст в чате (не замена JSON-фидбека
  │    (после фейла) │  валидатора; если вызов не удался — показывается validation_error)
  └────────┬─────────┘
           ▼
  Повтор ввода ответа или исчерпание попыток → следующий вопрос / результаты
           │
           ▼
  ┌──────────────────┐
  │ 5. COMPLETION    │  Один раз на экране результатов: тёплое завершающее сообщение
  └──────────────────┘


--------------------------------------------------------------------------------
ШАГ 1 — ГЕНЕРАЦИЯ ВОПРОСОВ (generate_questions)
--------------------------------------------------------------------------------
Назначение:
  По материалам урока (gift, extracted text, другие топики, предыдущие уроки)
  сгенерировать JSON: вопросы, допустимые ответы, баллы, min_pass_score,
  max_attempts, max_execution_time. Режим SHORT ANSWER или REASONING зависит
  от payload (topic_json.reasoningQuiz).

Где в коде:
  quiz_streamlit_app.py — build_system_prompt(), build_giftquiz_questions_prompt()
  или build_giftquiz_reasoning_questions_prompt().

Промпты:
  Зашиты в quiz_streamlit_app.py как PROMPT_SYSTEM_TEMPLATE,
  PROMPT_GIFTQUIZ_*_TEMPLATE (не вынесены в отдельный модуль).

Типовое значение модели по умолчанию:
  google/gemini-2.5-pro
Env:
  OPENROUTER_MODEL


--------------------------------------------------------------------------------
ШАГ 2 — РЕФРАЗЕР ВОПРОСА (rephrase_question)
--------------------------------------------------------------------------------
Назначение:
  Превратить «сухой» вопрос из JSON в естественную разговорную фразу; на первом
  вопросе — короткое интро к квизу из quiz_summary. После неудачной попытки
  может добавлять мягкое признание (not_passed) согласно промпту.

Файл промптов:
  quiz_question_rephraser.py
    - QUIZ_REPHRASER_SYSTEM_PROMPT
    - build_rephraser_user_prompt(...)

Вход в user-промпт:
  original_question, quiz_summary, is_first_question, language, not_passed,
  conversation_history (сообщения по всем вопросам до текущего включительно).

Кеш:
  Ключ: "{q_idx}:{len(conversation)}:{attempts}" — чтобы не вызывать LLM лишний раз
  при каждом rerun Streamlit.

Типовое значение модели по умолчанию:
  openai/gpt-5.1-chat-latest
Env:
  OPENROUTER_REPHRASE_MODEL

UI:
  На экране показывается рефразированный текст; сырой текст вопроса — в сворачиваемом блоке.


--------------------------------------------------------------------------------
ШАГ 3 — ВАЛИДАЦИЯ ОТВЕТА (validate_answer)
--------------------------------------------------------------------------------
Назначение:
  Оценить ответ с учётом истории диалога по текущему вопросу
  (combined answer), выставить validation_score [0..1], answer_score,
  классифицировать user_intent, заполнить validation_error если не пройден порог
  сложности (без прямой выдачи правильного ответа).

Файл промптов:
  quiz_answer_validation.py
    - QUIZ_ANSWER_VALIDATION_SYSTEM_PROMPT_BASE
    - build_answer_validation_system_prompt(validation_instruction)
        → добавляет блок сложности из DIFFICULTY_LEVELS в quiz_streamlit_app.py
    - build_answer_validation_user_prompt(...)

Выход:
  Строгий JSON с полями validation_score, answer_score, user_intent,
  reasoning, validation_error.

Типовое значение модели по умолчанию:
  openai/gpt-4o
Env:
  OPENROUTER_VALIDATION_MODEL

Дополнительно:
  К запросу может добавляться extra_body с reasoning (как в приложении).


--------------------------------------------------------------------------------
ШАГ 4 — FOLLOW-UP ПОСЛЕ НЕУДАЧНОЙ ПОПЫТКИ (followup_after_fail)
--------------------------------------------------------------------------------
Назначение:
  Сгенерировать короткий дружелюбный текст для чата после того, как валидатор не
  засчитал ответ. Промпт запрещает дословно цитировать validation_error и
  выдавать правильный ответ; учитывает intent, score, историю по текущему
  вопросу и по всему квизу.

Файл промптов:
  quiz_followup_question.py
    - QUIZ_FOLLOWUP_SYSTEM_PROMPT
    - followup_closeness_label(validation_score)
    - build_followup_user_prompt(...)

Вход:
  quiz_question, user_answer, validation_error, validation_score (как в промпте),
  current_question_messages, conversation_history_all_questions, language,
  user_intent.

Fallback:
  Если вызов не удался — в чат добавляется validation_error или "Try again!".

Типовое значение модели по умолчанию:
  openai/gpt-5.1-chat-latest
Env:
  OPENROUTER_FOLLOWUP_MODEL


--------------------------------------------------------------------------------
ШАГ 5 — СООБЩЕНИЕ О ЗАВЕРШЕНИИ (completion_message)
--------------------------------------------------------------------------------
Назначение:
  После подсчёта итогов квиза сгенерировать 2–3 предложения тёплого,
  естественного прощания на целевом языке (с учётом формальности для de/ru
  в системном промпте).

Файл промптов:
  quiz_completion_message.py
    - QUIZ_COMPLETION_SYSTEM_PROMPT
    - build_completion_user_prompt(quiz_summary, language)

Вход summary:
  Текст из контекста квиза + баллы, min pass, passed (см. _build_completion_summary
  в quiz_streamlit_app.py).

Сохранение:
  Результат кешируется в session_state.completion_message_for_session, чтобы
  не вызывать LLM повторно при rerun страницы результатов.

Типовое значение модели по умолчанию:
  google/gemini-3.1-flash-lite-preview
Env:
  OPENROUTER_COMPLETION_MODEL


--------------------------------------------------------------------------------
ПЕРЕМЕННЫЕ ОКРУЖЕНИЯ / СЕКРЕТЫ (кратко)
--------------------------------------------------------------------------------
  OPENROUTER_API_KEY          — обязательно
  OPENROUTER_MODEL            — шаг 1 (генерация)
  OPENROUTER_VALIDATION_MODEL — шаг 3
  OPENROUTER_REPHRASE_MODEL   — шаг 2
  OPENROUTER_FOLLOWUP_MODEL   — шаг 4
  OPENROUTER_COMPLETION_MODEL — шаг 5


--------------------------------------------------------------------------------
ЛОГИРОВАНИЕ
--------------------------------------------------------------------------------
Каждый успешный вызов OpenRouter в приложении добавляется в st.session_state.api_logs
с phase (например generate_questions, rephrase_q0, validate_q0_attempt1,
followup_q0_attempt1, completion_message). Лог можно скачать JSON из сайдбара
или с экрана результатов.
